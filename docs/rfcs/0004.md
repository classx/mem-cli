## Problem

Сейчас `mem-cli` — плоское хранилище: четыре таблицы (`facts`, `decisions`,
`commands`, `modules`), каждая запись это `content` + временные метки. Записи
**изолированы** и **не сгруппированы по темам**. Из этого следуют ограничения.

1. **Нет тематической группировки.** Контекст одной предметной области (напр.
   «auth») размазан по четырём сущностям: факт про auth в `facts`, решение в
   `decisions`, модуль в `modules`, команда в `commands`. Собрать «всё про auth»
   нельзя — только перечитать все четыре списка целиком.

2. **Восстановление контекста — всё или ничего.** Команды `list`/`decisions`/
   `modules` вываливают весь набор записей сущности. По мере роста (десятки
   записей) трудно дать агенту/человеку именно тот срез, что релевантен текущей
   задаче. Нет способа сказать «дай контекст вокруг темы X».

3. **Рост шума.** Чем больше записей, тем дороже (и по вниманию, и по токенам)
   читать всё подряд в начале сессии. Нужен дешёвый механизм сужения.

RFC-0003 предлагает решать это полноценным графом связей. Настоящий RFC —
**более лёгкая альтернатива**: теги. Граф решает большую задачу (направленные
типизированные отношения), но ценой заметной сложности и риска «пустого графа»
из-за высокого порога ведения связей. Для текущего масштаба `mem-cli` основная
реальная потребность — именно тематическая группировка («дай контекст по теме»),
которую теги закрывают минимальными средствами.

## Goal

Добавить **теги** — лёгкую тематическую разметку записей, чтобы:

- помечать любую запись (любой сущности) одним или несколькими тегами;
- собирать срез контекста по теме: «дай всё с тегом X» по всем сущностям сразу;
- сужать восстановление контекста в начале сессии до релевантной темы;

при этом:

- **не добавлять зависимостей** — теги моделируются в текущем SQLite;
- сохранить обратную совместимость (теги опциональны; существующие команды и
  данные не меняются);
- остаться в рамках минималистичной архитектуры (RFC-0001/0002) — заметно проще
  графа из RFC-0003.

Не-цели: направленные/типизированные связи (`supersedes`, `depends_on`),
транзитивные обходы, impact analysis — это область RFC-0003 (граф). Также вне
охвата: иерархии тегов, переименование/слияние тегов, синхронизация между
разработчиками.

## Design

### 1. Модель данных: одна таблица тегов

Узлы — существующие записи (`<entity>.id`). Тег — строка, связанная с записью.
Связь «многие-ко-многим» одной таблицей:

```sql
CREATE TABLE IF NOT EXISTS tags (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    entity     TEXT    NOT NULL,   -- 'facts' | 'decisions' | 'commands' | 'modules'
    record_id  INTEGER NOT NULL,
    tag        TEXT    NOT NULL,   -- нормализованный ярлык (см. §2)
    created_at TEXT    NOT NULL DEFAULT (datetime('now')),
    deleted_at TEXT,               -- soft-delete, как у записей
    UNIQUE (entity, record_id, tag)
);
CREATE INDEX IF NOT EXISTS idx_tags_tag    ON tags (tag);
CREATE INDEX IF NOT EXISTS idx_tags_record ON tags (entity, record_id);
```

Замечания по схеме:

- `entity` валидируется по существующему whitelist `ENTITY_TABLES`
  (`src/db.rs:209`) — тем же механизмом, что и сейчас, против SQL-инъекций
  (ср. тест `src/db.rs:431`).
- `UNIQUE (entity, record_id, tag)` запрещает дубль одного тега на записи.
- `deleted_at` — мягкое удаление, единообразно с записями сущностей.
- Внешнего ключа на `<entity>.record_id` нет (динамическое имя таблицы; FK в
  SQLite невозможен). Целостность — на уровне приложения: проверка существования
  записи при `tag` и каскад при удалении записи (см. §5).

Это **новая миграция v2** (текущая схема — v1, `MIGRATION_V1` в `src/db.rs`),
применяется через существующий механизм `schema_version` (`src/db.rs:170`).
Старые БД доедут до v2 автоматически (`IF NOT EXISTS`), данные не теряются.

### 2. Нормализация тегов

Чтобы `Auth`, `auth`, ` auth ` не плодили разные теги:

- trim пробелов по краям;
- приведение к нижнему регистру;
- пустой тег после нормализации — ошибка;
- допустимые символы — по whitelist-паттерну (напр. `[a-z0-9._-]+`); прочее —
  ошибка с понятным сообщением. Это упрощает поиск и исключает «грязные» строки.

Нормализация выполняется один раз при записи; в БД хранится уже нормализованный
тег.

### 3. CLI-команды

Добавляются поверх существующего интерфейса (`src/main.rs`, `enum Command`).
Существующие команды не меняются.

**Навесить тег(и):**
```sh
mem-cli tag <entity> <id> <tag>...        # один или несколько тегов за раз
# пример:
mem-cli tag decisions 3 auth security
```
- Проверяет существование записи (иначе ошибка).
- Нормализует и валидирует каждый тег (§2).
- Идемпотентно по `UNIQUE` (повторный тег — no-op с сообщением).

**Снять тег:**
```sh
mem-cli untag <entity> <id> <tag>         # soft-delete
mem-cli untag <entity> <id> <tag> --hard  # физическое удаление
```

**Найти записи по тегу (срез контекста по теме):**
```sh
mem-cli find <tag> [--entity <entity>] [--json]
# пример: весь контекст по теме auth
mem-cli find auth
```
- По умолчанию ищет по **всем** сущностям; группирует вывод по сущности.
- `--entity` сужает до одной сущности.
- `--json` — машиночитаемый вывод (как у `list`).

**Обзор тегов:**
```sh
mem-cli tags [--json]      # список всех тегов с количеством записей
```

**Диагностика целостности:**
```sh
mem-cli doctor [--json] [--fix]
```
Поскольку целостность тегов держится на уровне приложения (FK на динамическое
имя таблицы в SQLite невозможен, см. §1), нужен инструмент её проверки — иначе
рассинхрон копится незаметно. `doctor` проверяет:

- **висячие теги** — `record_id` указывает на несуществующую (purged) запись;
- **теги на soft-deleted записях** — запись помечена `deleted_at`;
- **«грязные» теги** — не прошли бы текущую нормализацию (§2); следы ручной
  правки БД или старых данных;
- **недопустимый `entity`** — вне whitelist `ENTITY_TABLES` (ручная правка БД).

Дополнительно (общая диагностика БД, полезна и без тегов): актуальность
`schema_version`, доступность БД на запись, счётчики записей и soft-deleted по
сущностям.

- `--json` — машиночитаемый отчёт.
- `--fix` — безопасная автопочинка: удаляет висячие теги и теги с недопустимым
  `entity`. По умолчанию (без `--fix`) — только отчёт, ничего не меняет. Код
  возврата: `0` если проблем нет, иначе ненулевой (удобно для CI/скриптов).

Замечание: ценность `doctor` пропорциональна сложности схемы — на исходной
плоской схеме проверять почти нечего, поэтому команда вводится **вместе** с
тегами (фичей, которая и порождает потребность в проверке целостности), а не
раньше.

### 4. Сценарии использования (для агента и человека)

- **Старт сессии, релевантный контекст по теме.** Вместо `list facts` +
  `decisions` + `modules` целиком — `mem-cli find auth`: собрать факты, решения,
  модули и команды темы одним срезом. Меньше шума, меньше токенов.
- **Разметка по ходу работы.** Дописывая факт/решение, сразу вешать тему:
  `mem-cli add facts "..."` → `mem-cli tag facts <id> auth`.
- **Обзор тем проекта.** `mem-cli tags` — какие предметные области вообще
  покрыты контекстом.

### 5. Целостность при удалении записей

При `delete`/`purge` записи её теги становятся «висячими». Поведение:

- `delete <entity> <id>` (soft): теги **не трогаем** — запись может быть
  восстановлена; `find`/`tags` по умолчанию скрывают теги soft-deleted записей.
- `purge <entity> <id>` (hard): каскадно удалить теги записи в одной транзакции
  с удалением записи.

Требует точечной правки `cmd_delete`/`cmd_purge` (`src/main.rs:251,264`).

### 6. Объём изменений в коде

- `src/db.rs`: миграция v2 (таблица `tags` + индексы); функции `tag`, `untag`,
  `find`, `tags`; нормализация/валидация тега; каскад в `purge`; функция
  диагностики целостности для `doctor` (+ опциональная починка). Юнит-тесты по
  образцу существующих (включая защиту от инъекций в `entity`, «грязные» теги и
  обнаружение/починку висячих тегов).
- `src/main.rs`: новые варианты `enum Command` (`Tag`, `Untag`, `Find`, `Tags`,
  `Doctor`) и обработчики `cmd_*`.
- Документация: `README.md` (раздел «Возможности» и примеры), `AGENTS.md` (как
  агенту использовать `find <tag>` для восстановления контекста по теме),
  `CHANGELOG.md`, bump версии (minor — фича аддитивная, обратно совместимая).

## Преимущества и недостатки

### Преимущества

- **Радикальная простота.** Одна таблица, четыре команды, плоские запросы. Нет
  направлений, типов связей, рекурсивных обходов. Ближе всего к девизу
  «минимальный CLI» (RFC-0001).
- **Низкий порог ведения.** Повесить тег `auth` проще, чем продумывать
  направленное типизированное ребро. Выше шанс, что разметкой реально будут
  пользоваться (главный риск графа — пустой граф из-за высокого порога).
- **Прямое попадание в основной сценарий.** «Дай контекст по теме X» —
  вероятно, самый частый реальный запрос; теги закрывают его естественно.
- **Без новых зависимостей; обратная совместимость.** Теги опциональны;
  существующие команды, данные и тесты не меняются.
- **Переиспользование инфраструктуры.** Whitelist таблиц, soft-delete,
  `schema_version`-миграции, JSON-вывод — всё уже есть.
- **Меньше кода и багов целостности.** Нет «висячих рёбер» с двумя концами —
  тег это просто строка у одной записи.

### Недостатки и риски

- **Нет направленных отношений.** Не выразить `supersedes` (решение заменяет
  решение) и `depends_on` (модуль зависит от модуля) — это асимметричные связи.
  История решений и impact-analysis недоступны (область RFC-0003).
- **Нет транзитивности.** «Что сломается на 3 уровня вглубь» — невозможно. Теги
  это плоская группировка, не цепочки.
- **Размытая семантика связи.** Общий тег не говорит, *как* записи связаны
  (обосновывает? отменяет? просто рядом?). Граф несёт тип в ребре.
- **Ложные/шумные связи.** Одинаковый тег ≠ настоящая связь; при росте крупная
  тема собирает много записей в кучу — снова «дамп», только по теме.
- **Ручная дисциплина.** Теги не появляются сами; без привычки помечать записи
  срез по теме будет неполным. (Риск ниже, чем у графа, но он есть.)

## Alternatives

1. **Граф связей (RFC-0003).** Плюс: направленные типизированные отношения,
   транзитивные обходы, impact analysis, история решений. Минус: заметно сложнее
   (рёбра, каскады, рекурсивные CTE, +команды) и выше порог ведения → риск
   пустого графа. Решает большую задачу, чем реально нужна на текущем масштабе.
   Теги и граф **не взаимоисключающи**: можно начать с тегов, а граф добавить
   позже узким RFC, если потребность в направленных связях подтвердится.

2. **Полноценная / встраиваемая graph DB (Neo4j, KùzuDB).** Плюс: настоящие
   графовые запросы. Минус: тяжёлая внешняя зависимость, противоречит
   «минимальный CLI на Rust + SQLite» и правилу о зависимостях; оверкилл для
   личной локальной БД. **Отклонено** (см. также RFC-0003 Alternatives).

3. **Теги в свободном тексте `content`** (напр. `#auth` внутри записи). Плюс:
   ноль изменений схемы. Минус: не нормализовано, хрупкий поиск подстрокой, нет
   обзора тегов, нет целостности. **Отклонено.**

4. **Категория-перечисление вместо свободных тегов** (фиксированный enum тем).
   Плюс: контролируемый словарь, нет «грязных» тегов. Минус: темы проекта заранее
   неизвестны и меняются; постоянные правки enum в коде. Свободные теги с
   нормализацией (§2) — гибче. **Отклонено**, но whitelist-паттерн символов
   оставляем как компромисс.

5. **Ничего не делать.** Плюс: сохраняем минимализм; для малых объёмов плоский
   список достаточен. Минус: не решает §Problem. Валидно, **если** тематический
   срез на практике не нужен — первый вопрос review.

## Voting

Открытые вопросы для review (заполняется по ходу обсуждения):

1. **Теги или граф (RFC-0003)** как первый шаг — или оба последовательно (теги
   сейчас, граф позже при подтверждённой потребности)?
2. **Свободные теги vs фиксированный словарь** (§2 vs Alternative 4).
3. **Паттерн допустимых символов** тега (`[a-z0-9._-]+`?) и регистр.
4. **Поведение каскада при `purge`** (§5): hard-delete тегов или soft-delete.
5. **Объём v1:** включать ли `mem-cli tags` (обзор) сразу или отложить.
6. **`doctor --fix`** (§3): автопочинка безопасна по умолчанию или требует явного
   подтверждения; чинить только висячие теги или и «грязные» тоже.

## Migration

- **Схема:** новая миграция v2 (таблица `tags` + индексы) применяется к
  существующим БД через `schema_version` (`src/db.rs:170`). Идемпотентна
  (`IF NOT EXISTS`); данные сущностей не затрагиваются.
- **Данные:** обратной заливки нет — существующие записи остаются без тегов;
  теги добавляются по мере использования `tag`.
- **Совместимость:** все текущие команды и их вывод неизменны; фича аддитивная.
- **Версия:** minor-bump в `Cargo.toml` + запись в `CHANGELOG.md` (English),
  согласно git-workflow проекта. Не breaking.

## Связь с RFC-0003

RFC-0003 (граф) и настоящий RFC решают пересекающуюся проблему (изоляция записей,
дамп-восстановление контекста) разными средствами. Они **конкурирующие как
первый шаг**, но **совместимые в перспективе**: теги дают тематическую
группировку дёшево; граф добавляет направленные связи дорого. Рекомендуемая
фазировка для обсуждения: внедрить теги, а граф рассматривать позже только под
конкретные сценарии `supersedes`/`depends_on`. Если на review выбран граф —
настоящий RFC может быть помечен `superseded_by: 3`; если выбраны теги —
зеркально в RFC-0003.
