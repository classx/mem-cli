## Problem

`mem-cli` уже решает хранение и извлечение проектного контекста локально
(SQLite + CLI), но интеграция с агентами/IDE сегодня строится через вызов
shell-команд (`mem-cli list facts`, `mem-cli find auth` и т.д.). Это работает,
но имеет ограничения:

1. **Нет стандартного tool-интерфейса для LLM-клиентов.** Каждый клиент должен
   сам парсить CLI-вывод и договариваться о форматах.
2. **Низкая композиционность в экосистеме MCP.** Современные агенты ожидают
   инструменты и ресурсы по Model Context Protocol, а не ad-hoc shell wrapping.
3. **Лишняя связка «текстовый вывод → повторный парсинг».** При наличии
   `--json` мы всё равно гоняем внешнюю команду, вместо прямого typed tool API.

Нужен стандартный программный интерфейс к уже существующим возможностям
`mem-cli`, без отказа от CLI как основного UX для разработчика.

## Goal

Добавить **MCP-сервер** в `mem-cli`, чтобы:

- предоставлять операции контекста как MCP tools/resources;
- сохранить обратную совместимость: CLI остаётся рабочим и поддерживаемым;
- не ломать текущую модель хранения (локальная SQLite БД, правила резолва пути).

Не-цели:

- сетевой multi-user сервис и удалённая синхронизация;
- смена СУБД или архитектуры хранения;
- переработка бизнес-логики команд `add/list/find/tag/...`.

## Design

### 1. Общий core-слой и два frontend’а

Текущую доменную логику хранения оставляем в существующем слое (`src/db.rs`).
Добавляется второй frontend к тому же core:

- **CLI frontend** (существующий, `src/main.rs`);
- **MCP frontend** (новый модуль, transport: `stdio`).

Принцип: MCP-слой не дублирует SQL/валидацию, а вызывает те же функции, что CLI.

### 2. Новая подкоманда CLI

Добавляется команда:

```sh
mem-cli mcp
```

Команда запускает MCP-сервер по `stdio`, чтобы клиент (Copilot, Claude Desktop,
и др.) мог подключаться как к локальному tool-provider.

### 3. Поверхность MCP API

#### Tools (минимальный v1)

1. `list_facts`
2. `list_decisions`
3. `list_modules`
4. `list_commands`
5. `find_by_tag` (`tag`)
6. `add_fact`, `add_decision`, `add_module`, `add_command`
7. `tag_record` / `untag_record`
8. `doctor`

Имена можно уточнить в реализации, но принцип остаётся: один MCP tool = один
устойчивый use-case текущего CLI.

#### Resources (опционально в v1, обязательно в v2)

- `mem://facts`
- `mem://decisions`
- `mem://modules`
- `mem://commands`

Ресурсы читаемые, отражают актуальные данные БД и позволяют агентам брать
срезы контекста без вызова tool при простом чтении.

### 4. Формат входа/выхода и ошибки

- Вход tools: строго типизированные JSON-схемы (entity, id, tag и т.д.).
- Выход: JSON-объекты без табличного форматирования.
- Ошибки: явные, без silent fallback, с читаемыми сообщениями для клиента.
- Поведение должно соответствовать CLI (одинаковая валидация и ограничения).

### 5. Безопасность и границы

- MCP-сервер работает только с локальной БД, резолв пути неизменен:
  `MEMORY_DB_DIR` → `.mem-project` → `.memory/`.
- Выполнение произвольного SQL или shell через MCP не поддерживается.
- Доступ ограничен операциями, эквивалентными текущим командам `mem-cli`.

### 6. Наблюдаемость и DX

- Логи сервера — в stderr (протокол в stdout не загрязняется).
- Для отладки добавить smoke-команду проверки запуска (`mem-cli mcp --help` и
  интеграционный тест старта/остановки процесса).

### 7. План внедрения

1. Вынести/дочистить общие обработчики, чтобы CLI и MCP вызывали один core.
2. Реализовать MCP server + минимальный набор tools (v1).
3. Добавить ресурсы и расширенные tools (v2, при подтверждённой потребности).
4. Обновить `README.md` и `AGENTS.md` примерами подключения MCP-клиентов.

## Преимущества и недостатки

### Преимущества

- Стандартная интеграция с MCP-экосистемой без shell-обвязок.
- Переиспользование существующей логики и данных.
- Снижение хрупкости парсинга CLI-вывода в агентах.

### Недостатки и риски

- Увеличение поверхности поддержки (CLI + MCP).
- Потребуются интеграционные тесты протокольного поведения.
- Нужен аккуратный контроль совместимости схем tool-вызовов при релизах.

## Alternatives

1. **Оставить только CLI (status quo).**
   Плюсы: нулевая сложность. Минусы: отсутствие стандартного MCP-интерфейса.

2. **Отдельный внешний MCP-прокси-репозиторий.**
   Плюсы: изоляция протокольного кода. Минусы: дублирование логики/версий и
   расхождение с эволюцией `mem-cli`.

3. **HTTP/REST вместо MCP.**
   Плюсы: привычный API. Минусы: не решает задачу нативной интеграции с
   MCP-клиентами и добавляет лишний transport-слой.

## Migration

- **Схема БД:** без изменений.
- **CLI-совместимость:** полная, существующие команды не меняются.
- **Документация:** добавить разделы по запуску MCP и примеры конфигурации
  клиентов.
- **Версия:** minor bump (аддитивная функциональность, без breaking changes).
