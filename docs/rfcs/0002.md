## Problem

Сейчас БД (`project_context.db`) по умолчанию создаётся в каталоге `./.memory/`
относительно текущего рабочего каталога (`db_dir()` возвращает `MEMORY_DB_DIR`,
иначе `./.memory`). Это приводит к ряду проблем:

1. **Бинарный SQLite в гит-репозитории.** Если каталог попадает под версионный
   контроль, получаем бесполезные диффы, неразрешимые merge-конфликты и шум в
   истории.
2. **Смешение персонального и общего.** Факты у разных разработчиков свои, и
   написаны они могут быть на разных языках. Общий репозиторий — неподходящее
   место для персонального контекста.
3. **Зависимость от cwd.** `./.memory/` резолвится относительно текущего
   каталога: запуск `mem-cli` из подкаталога проекта создаёт/читает другую БД.
   Это скрытый источник ошибок уже сейчас.

## Goal

Хранить персональную БД контекста вне репозитория, в домашнем каталоге
пользователя, при этом:

- надёжно сопоставлять рабочий проект с его БД из любого подкаталога;
- не ломать существующее поведение (`MEMORY_DB_DIR`, тесты в `src/db.rs`);
- объяснить агенту (через `AGENTS.md`), где искать контекст.

## Design

### 1. Идентификатор проекта в репозитории, данные — в `$HOME`

Развести две сущности:

- **Идентификатор проекта** — маленький текстовый маркер, который *можно*
  коммитить (общий, не бинарный, без конфликтов). Хранит слаг/имя проекта.
- **Данные** — личная SQLite-БД в домашнем каталоге пользователя.

### 2. `mem-cli init <NAME>`

- Принимает имя проекта `NAME` (опционально; при отсутствии — производное от
  имени каталога проекта).
- Вычисляет каталог:
  `${XDG_DATA_HOME:-~/.local/share}/mem/<slug>/project_context.db`.
- `<slug>` = безопасное для ФС преобразование `NAME` + случайный id, например
  `my-proj-a1b2c3`. Случайный id генерируется **один раз** при `init`,
  сохраняется в маркере `.mem-project` и далее неизменен независимо от пути,
  машины и реклонов. Это гарантирует уникальность между разными репозиториями с
  одинаковым `NAME` без привязки к (нестабильному) пути или git-remote.
  Преобразование `NAME` важно для русских имён, пробелов и слэшей.
- Создаёт маркер-файл `.mem-project` в корне репозитория со слагом и дополняет
  `AGENTS.md` пояснением, где хранится контекст (без абсолютного пути — `$HOME` у
  всех разный; хранить только `NAME`/слаг).

### 3. Разрешение БД для `add` / `list` / `decisions` / `modules` / ...

Ключевая задача — *discovery*: как остальные команды узнают проект без передачи
`NAME` каждый раз. Решение: поиск маркера `.mem-project` вверх по дереву
каталогов от cwd (как делает `git`), затем `slug -> ~/.local/share/mem/<slug>/`.

Приоритет источников пути (от высшего к низшему):

1. `MEMORY_DB_DIR` (или новый флаг `--db <PATH>`) — override для тестов, CI и
   ручного использования. Сохраняет зелёные тесты в `src/db.rs`.
2. Маркер `.mem-project`, найденный вверх по дереву от cwd.
3. Дефолт (см. ниже про обратную совместимость).

### 4. Маркер проекта

Выбран отдельный файл `.mem-project` в корне репозитория, хранящий полный слаг
(`<NAME-slug>-<random-id>`). Текстовый, крошечный, без бинарных конфликтов;
находится поиском вверх по дереву. Случайный id генерируется один раз при `init`
и фиксируется здесь, поэтому слаг неизменен при переносе/реклоне репозитория.
Может коммититься (общий, безопасный) — он указывает лишь слаг, а не личные
данные.

### 5. Обратная совместимость и переход

- Сохранить поддержку `MEMORY_DB_DIR` с высшим приоритетом.
- Если существует старый `./.memory/project_context.db` — поддержать его чтение
  и/или предложить миграцию в новый путь.
- `AGENTS.md` в этом репозитории уже существует — дополнять, не перезаписывать.

## Alternatives

1. **Оставить `./.memory/` + `.gitignore`.** Плюс: минимум изменений. Минус: не
   решает зависимость от cwd; легко закоммитить случайно; данные привязаны к
   рабочей копии (потеря при реклоне).
2. **Хранить в `~/.config/mem/<NAME>` (исходное предложение).** Плюс: просто.
   Минус: `~/.config` — для конфигурации, не для растущих данных; хардкод
   игнорирует `XDG_*`; коллизии имён между разными репозиториями с одинаковым
   `NAME`; не решён discovery для последующих команд.
3. **Требовать `NAME` в каждой команде.** Плюс: явно. Минус: плохой UX,
   ошибки из-за опечаток.
4. **Привязка к git-remote/корню репозитория вместо имени.** Плюс: нет коллизий,
   нет ручного `NAME`. Минус: «магия», не работает вне git-репозитория.

## Voting

Решения, принятые в ходе review:

1. **Базовый каталог данных:** `XDG_DATA_HOME` (`~/.local/share/mem/<slug>/`) —
   как стандарт для растущих данных (не `~/.config`).
2. **Механизм discovery:** маркер-файл `.mem-project` в корне репозитория со
   слагом; поиск вверх по дереву от cwd.
3. **Идентичность проекта:** `slug = преобразованный NAME + случайный id`,
   сгенерированный один раз при `init` и сохранённый в `.mem-project` (напр.
   `my-proj-a1b2c3`). Неизменен при переносе/реклоне; исключает коллизии между
   разными репозиториями с одинаковым именем без привязки к нестабильному пути
   или git-remote.
4. **Override:** `MEMORY_DB_DIR` сохраняется с наивысшим приоритетом (не ломает
   тесты в `src/db.rs` и CI); опционально — флаг `--db <PATH>`.
5. **Версия:** смена дефолтного пути — потенциально breaking; нужен переходный
   период с поддержкой чтения старых `./.memory/` и подсказкой о миграции.

## Migration

- Старые БД в `./.memory/` продолжают читаться; предусмотреть команду/подсказку
  для переноса в новый путь.
- `MEMORY_DB_DIR` сохраняет наивысший приоритет (не ломает тесты и CI).
- Смена дефолтного расположения отражается в `CHANGELOG.md` и версии (минимум
  minor; при breaking-изменении дефолта — major или явный переходный режим).
