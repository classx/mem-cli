## Problem

Сейчас `mem-cli` — плоское хранилище: четыре таблицы (`facts`, `decisions`,
`commands`, `modules`), каждая запись это `content` + временные метки. Записи
**изолированы**: между ними нет связей. Из этого следует ряд ограничений.

1. **Нет контекста «что с чем связано».** Невозможно ответить на вопросы
   «какие решения затрагивают модуль X?», «какая команда относится к модулю Y?»,
   «какой факт обосновывает это решение?». Связь существует только в голове
   разработчика или в тексте записи (в свободной форме, не машиночитаемо).

2. **Нет истории/преемственности решений.** Когда новое решение отменяет старое,
   единственный способ — soft-delete старого (`deleted_at`) или ручная пометка в
   тексте. Цепочка «решение A заменено решением B заменено решением C» (классика
   ADR `supersedes`) не выражается.

3. **Восстановление контекста — всё или ничего.** Команды `list`/`decisions`/
   `modules` вываливают весь набор записей сущности. По мере роста (десятки
   записей) агенту/человеку трудно понять, что релевантно конкретной задаче.
   Нет способа сказать «дай контекст вокруг модуля X».

4. **Нет анализа влияния (impact analysis).** «Если я меняю модуль A, что ещё
   затронется?» требует обхода зависимостей `module —[depends_on]→ module`,
   которого сейчас нет.

Вопрос, который инициировал этот RFC: «существуют graph DB — как это может помочь
проекту?». Ниже разбираем, какую *именно* пользу даёт графовая модель и **как**
её внедрить, не нарушая дух проекта (минимальный CLI на Rust + SQLite, без лишних
зависимостей).

## Goal

Добавить в `mem-cli` **связи между записями как первоклассную сущность**, чтобы:

- выражать отношения: `affects`, `depends_on`, `supersedes`, `relates_to`,
  `justifies` и т.п.;
- отвечать на реляционные и транзитивные запросы («что затрагивает модуль X»,
  «что устарело после решения Y», «дай связанный контекст»);
- восстанавливать контекст *по релевантности*, а не дампом всех записей;

при этом:

- **не добавлять графовую СУБД** (Neo4j и пр.) и вообще новых внешних
  зависимостей — граф моделируется в текущем SQLite;
- сохранить обратную совместимость (существующие команды и данные работают
  без изменений; связи — опциональны);
- остаться в рамках минималистичной архитектуры (RFC-0001/0002).

Не-цели: полнотекстовый/семантический поиск, эмбеддинги, визуализация графа,
синхронизация между разработчиками.

## Design

### 1. Модель данных: одна таблица рёбер

Граф моделируется поверх существующих таблиц одной таблицей рёбер. Узлы графа —
существующие записи (`<entity>.id`); рёбра — строки новой таблицы `edges`.

```sql
CREATE TABLE IF NOT EXISTS edges (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    from_entity TEXT    NOT NULL,   -- 'facts' | 'decisions' | 'commands' | 'modules'
    from_id     INTEGER NOT NULL,
    to_entity   TEXT    NOT NULL,
    to_id       INTEGER NOT NULL,
    relation    TEXT    NOT NULL,   -- 'affects' | 'depends_on' | 'supersedes' | ...
    created_at  TEXT    NOT NULL DEFAULT (datetime('now')),
    deleted_at  TEXT,               -- soft-delete, как у записей
    UNIQUE (from_entity, from_id, to_entity, to_id, relation)
);
CREATE INDEX IF NOT EXISTS idx_edges_from ON edges (from_entity, from_id);
CREATE INDEX IF NOT EXISTS idx_edges_to   ON edges (to_entity, to_id);
```

Замечания по схеме:

- `from_entity`/`to_entity` валидируются по существующему whitelist
  `ENTITY_TABLES` (`src/db.rs:209`) — тем же механизмом, что и сейчас, чтобы не
  допустить SQL-инъекций (ср. тест `src/db.rs:431`).
- `relation` валидируется по whitelist допустимых типов связей (см. §3) —
  иначе граф «расползётся» в неконтролируемый набор строк.
- `UNIQUE` запрещает дубли одного и того же ребра.
- `deleted_at` — мягкое удаление, единообразно с записями сущностей.
- Внешних ключей на `<entity>.id` нет (записи в разных таблицах; FK на
  динамическое имя таблицы в SQLite невозможен). Целостность поддерживается на
  уровне приложения при создании ребра (проверка существования обоих концов) и
  каскадом при удалении узла (см. §5).

Это **новая миграция v2** (текущая схема — v1, `MIGRATION_V1` в `src/db.rs`).
Применяется к существующим БД через механизм `schema_version`, уже имеющийся в
коде (`src/db.rs:170`). Старые БД доедут до v2 автоматически, данные не теряются.

### 2. Направленность и семантика рёбер

Рёбра **направленные**: `from —[relation]→ to`. Семантика типов (предлагаемый
стартовый набор):

| relation     | from → to                | смысл                                   |
|--------------|--------------------------|-----------------------------------------|
| `affects`    | decision → module        | решение затрагивает модуль              |
| `depends_on` | module → module          | модуль зависит от модуля                |
| `supersedes` | decision → decision      | решение заменяет предыдущее             |
| `justifies`  | fact → decision          | факт обосновывает решение               |
| `relates_to` | любой → любой            | общая нетипизированная связь            |

Набор намеренно небольшой и **расширяемый через whitelist** в коде. `relates_to`
— «клапан» для случаев, не покрытых специализированными типами, чтобы не плодить
типы преждевременно.

Запросы учитывают направление: «что затрагивает модуль M» = рёбра с
`to=(modules, M)` и `relation=affects`. Для удобства команда `related` (см. §4)
по умолчанию показывает связи в **обе** стороны.

### 3. CLI-команды

Добавляются команды поверх существующего интерфейса (`src/main.rs`, `enum
Command`). Существующие команды не меняются.

**Создание связи:**
```sh
mem-cli link <from_entity> <from_id> <to_entity> <to_id> --rel <relation>
# пример:
mem-cli link decisions 3 modules 1 --rel affects
```
- Проверяет существование обоих концов (иначе ошибка).
- Проверяет `relation` по whitelist.
- Идемпотентно по `UNIQUE` (повторный `link` — ошибка либо no-op с сообщением).

**Удаление связи:**
```sh
mem-cli unlink <edge_id>            # по id ребра (soft-delete)
mem-cli unlink <edge_id> --hard     # физическое удаление
```

**Просмотр связей записи:**
```sh
mem-cli related <entity> <id> [--rel <relation>] [--depth N] [--json]
# пример: что связано с модулем 1
mem-cli related modules 1
```
- По умолчанию `--depth 1` (прямые соседи в обе стороны).
- `--depth N>1` — транзитивный обход (рекурсивный CTE, см. §6) с защитой от
  циклов.
- `--rel` фильтрует по типу связи.
- `--json` — машиночитаемый вывод (как у `list`).

**Список всех рёбер (для отладки/обзора):**
```sh
mem-cli edges [--json]
```

### 4. Сценарии использования (для агента и человека)

- **Старт сессии, релевантный контекст.** Вместо `list facts` + `decisions` +
  `modules` целиком — взять модуль, который трогаешь, и `mem-cli related modules
  1 --depth 2`: получить связанные решения, обосновывающие факты и команды.
- **Impact analysis.** `mem-cli related modules A --rel depends_on --depth 3` —
  что сломается при изменении A.
- **История решений.** Цепочка `supersedes` показывает эволюцию: текущее решение
  и что оно заменило. `deleted_at` при этом можно не использовать — старое
  решение остаётся видимым как звено цепочки.

### 5. Целостность при удалении узлов

При `delete`/`purge` записи сущности её рёбра становятся «висячими». Поведение:

- `delete <entity> <id>` (soft): рёбра **не трогаем** — запись может быть
  восстановлена; `related` по умолчанию скрывает рёбра к soft-deleted записям.
- `purge <entity> <id>` (hard): каскадно soft-delete или hard-delete рёбер, у
  которых этот узел — конец. Каскад выполняется в одной транзакции с удалением
  записи.

Это требует точечной правки `cmd_delete`/`cmd_purge` (`src/main.rs:251,264`).

### 6. Транзитивные запросы

`--depth N>1` реализуется рекурсивным CTE по `edges` (SQLite поддерживает
`WITH RECURSIVE`). Защита от циклов — отслеживание посещённых узлов в пути
(строка-аккумулятор `visited`) и ограничение глубины. Это «80% пользы графа» без
графового движка: для личной БД с десятками–сотнями записей производительности
SQLite + индексов более чем достаточно.

### 7. Объём изменений в коде

- `src/db.rs`: миграция v2 (таблица `edges` + индексы); функции `link`,
  `unlink`, `related`, `edges`, валидация `relation` по whitelist; каскад в
  `purge`. Юнит-тесты по образцу существующих (включая защиту от инъекций в
  `relation`/`entity`).
- `src/main.rs`: новые варианты `enum Command` (`Link`, `Unlink`, `Related`,
  `Edges`) и их обработчики `cmd_*`.
- Документация: `README.md` (раздел «Возможности» и примеры), `AGENTS.md`
  (как агенту использовать `related` для восстановления контекста), `CHANGELOG.md`,
  bump версии (minor — фича аддитивная, обратно совместимая).

## Преимущества и недостатки

### Преимущества

- **Выразительность.** Появляются отношения, которых принципиально не было:
  impact analysis, история решений, релевантный контекст.
- **Без новых зависимостей.** Граф живёт в текущем SQLite; не нарушает дух
  проекта (RFC-0001) и правило «не добавлять зависимости без причины».
- **Обратная совместимость.** Связи опциональны; существующие команды, данные и
  тесты не меняются. Не пользуешься `link` — поведение прежнее.
- **Переиспользование инфраструктуры.** Whitelist таблиц, soft-delete,
  `schema_version`-миграции, JSON-вывод — всё уже есть.
- **Контролируемый граф.** Whitelist типов связей не даёт схеме «расползтись».

### Недостатки и риски

- **Рост сложности CLI и кода.** +4 команды, новая таблица, каскады, рекурсивные
  CTE. Для проекта, девиз которого «минимальный CLI», это заметный шаг.
- **Ручное ведение связей.** Рёбра не появляются сами — их нужно создавать
  `link`. Если на практике дисциплины нет, граф будет пустым/неполным и пользы
  не даст. **Ключевой риск окупаемости.**
- **Целостность на уровне приложения.** Нет FK на динамические таблицы → «висячие
  рёбра» при удалении узлов нужно обрабатывать вручную (§5); риск рассинхрона.
- **Возможный оверинжиниринг.** Если реальный объём контекста — единицы записей и
  старт сессии = «прочитать всё», граф не окупится. Польза начинается на десятках
  записей.
- **Не настоящая graph DB.** Нет графовых обходов уровня Cypher, паттерн-матчинга,
  оптимизатора под графы. Для текущих масштабов это не нужно, но потолок ниже.

## Alternatives

1. **Полноценная graph DB (Neo4j, Dgraph, KùzuDB).** Плюс: настоящие графовые
   запросы (Cypher), обходы, паттерн-матчинг. Минус: тяжёлая внешняя зависимость
   (сервер/встраиваемый движок), противоречит «минимальный CLI на Rust + SQLite»
   и правилу о зависимостях; оверкилл для личной локальной БД на десятки записей.
   **Отклонено.**

2. **Встраиваемый графовый движок в процессе (напр. KùzuDB как Rust-крейт).**
   Плюс: без отдельного сервера, Cypher-подобные запросы. Минус: всё ещё крупная
   новая зависимость и второй формат хранения рядом с SQLite; дублирование
   инфраструктуры миграций/soft-delete. **Отклонено** в пользу «граф в SQLite».

3. **Связи в свободном тексте `content`.** Плюс: ноль изменений схемы. Минус: не
   машиночитаемо, нет запросов/обходов, нет целостности — то, что есть сейчас, и
   что мы хотим устранить. **Отклонено.**

4. **Теги/категории вместо рёбер** (поле `tags` у записей). Плюс: проще рёбер;
   группировка «вокруг темы». Минус: не выражает *направленные* отношения
   (`supersedes`, `depends_on`), нет транзитивности, нет impact analysis.
   Решает меньшую задачу. Может быть отдельным более лёгким RFC, если граф
   признают избыточным.

5. **Ничего не делать.** Плюс: сохраняем минимализм; для малых объёмов контекста
   плоский список достаточен. Минус: не решает ни одну из проблем §Problem.
   Валидная опция, **если** связи на практике не нужны — поэтому первый вопрос
   review: подтверждаем ли мы потребность.

## Voting

Открытые вопросы для review (заполняется по ходу обсуждения):

1. **Нужны ли связи вообще** при текущем объёме контекста, или плоского списка
   достаточно? (Главный вопрос окупаемости; ср. Alternative 5.)
2. **Стартовый набор типов связей** (§2): достаточно ли пяти; нужен ли
   `relates_to`-«клапан»; разрешать ли произвольные типы.
3. **Поведение каскада при `purge`** (§5): soft-delete рёбер или hard-delete.
4. **Глубина обхода по умолчанию** в `related`: 1 или 2.
5. **Объём v1 фичи:** реализовать сразу `--depth N` (рекурсивный CTE) или начать
   с `--depth 1` и отложить транзитивность.

## Migration

- **Схема:** новая миграция v2 (таблица `edges` + индексы) применяется к
  существующим БД через `schema_version` (`src/db.rs:170`). Идемпотентна
  (`IF NOT EXISTS`); данные сущностей не затрагиваются.
- **Данные:** обратной заливки нет — существующие записи остаются без рёбер;
  связи добавляются по мере использования `link`.
- **Совместимость:** все текущие команды и их вывод неизменны; фича аддитивная.
- **Версия:** minor-bump в `Cargo.toml` + запись в `CHANGELOG.md` (English),
  согласно git-workflow проекта. Не breaking.
