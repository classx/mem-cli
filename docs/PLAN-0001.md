# PLAN-0001: Проект Rust + SQLite (mem-cli)

Реализация [RFC-0001](rfcs/0001.md). План разбит на фазы; каждая фаза —
отдельный branch, каждая задача — sub-branch `phase-N/X.Y-description`.

## Фаза 1 — Каркас проекта

- **1.1** Инициализировать Cargo-проект (`cargo init --name mem-cli`), `.gitignore`
  (исключить `target/`, `*.db`).
- **1.2** Подобрать зависимости: `rusqlite` (SQLite), `clap` (разбор CLI),
  `anyhow`/`thiserror` (ошибки). Минимальный набор, без лишнего.
- **1.3** Точка входа `main.rs`: каркас CLI с подкомандами-заглушками.
- **1.4** Настроить базовый CI-набор команд: `cargo build`, `cargo test`,
  `cargo clippy`, `cargo fmt`.

## Фаза 2 — Слой хранения и схема

- **2.1** Модуль `db`: открытие/создание `project_context.db`.
  Каталог БД: если задана переменная окружения `MEMORY_DB_DIR` — использовать её,
  иначе `.memory/`. Файл БД — `<dir>/project_context.db`
  (каталог создаётся при необходимости).
- **2.2** Таблица `schema_version` + механизм версионирования/миграций.
- **2.3** Схема сущностей с полем `deleted_at` (для soft delete):
  - `facts` — стабильные факты о проекте;
  - `decisions` — принятые технические решения;
  - `commands` — проверенные команды build/test/lint;
  - `modules` — краткое описание модулей и их ответственности.
- **2.4** Общие поля: `id`, `created_at`, `updated_at`, `deleted_at`.

## Фаза 3 — Команда `init` (реализована вместе с Фазой 2)

- **3.1** `mem-cli init` — создать БД и применить схему (идемпотентно).
- **3.2** Записать текущую версию схемы в `schema_version`.
- **3.3** Тесты: создание БД, повторный вызов не ломает данные.

> Примечание: `init` реализован в ветке `phase-2`, т.к. является
> естественным потребителем db-слоя (иначе модуль `db` — мёртвый код
> для `clippy -D warnings`).

## Фаза 4 — Универсальные действия `add` / `list`

- **4.1** `mem-cli add <facts|decisions|commands|modules> ...` — добавление записи.
- **4.2** `mem-cli list <facts|decisions|commands|modules>` — вывод записей
  (по умолчанию скрывать `deleted_at`). Формат: таблица по умолчанию,
  JSON по флагу `--json`.
- **4.3** Единый слой работы с сущностями, чтобы избежать дублирования кода.
- **4.4** Тесты на add/list для каждой сущности.

## Фаза 5 — Специализированные режимы

- **5.1** `mem-cli decisions` — быстрый доступ к `decisions` (алиас list).
- **5.2** `mem-cli modules` — быстрый доступ к `modules` (алиас list).

## Фаза 6 — Удаление (soft / hard)

- **6.1** `mem-cli delete <entity> <id>` — выставить `deleted_at`.
- **6.2** `mem-cli purge <entity> <id>` — физическое удаление строки.
- **6.3** Тесты: delete скрывает запись из `list`, purge удаляет физически.

## Принципы реализации

- Основной интерфейс к данным — `mem-cli`; прямой SQL только для отладки.
- Минимальные изменения, без новых зависимостей без явной причины.
- `main`/`master` всегда зелёный (build + test + lint).
- Каждый commit с trailer `Co-authored-by: Copilot <223556219+Copilot@users.noreply.github.com>`.

## Решения по открытым вопросам

- Расположение БД: каталог из env-переменной `MEMORY_DB_DIR`, иначе `.memory/`;
  файл — `<dir>/project_context.db`.
- Формат вывода `list`: таблица по умолчанию, JSON по флагу `--json`.
